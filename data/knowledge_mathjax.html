A Vol Overlay team typically sits above individual books and optimizes firm-level volatility risk while seeking risk-adjusted alpha.

4) Greeks Refresher (with question Intuition)

Let option value be $V(S,\sigma,t,...)$.

  • Delta: $\Delta=\partial V/\partial S$
  • Directional sensitivity to spot.
  • Gamma: $\Gamma=\partial^2 V/\partial S^2$
  • Convexity; sensitivity of delta to spot changes.
  • Vega: $\nu=\partial V/\partial \sigma$
  • Sensitivity to implied vol level.
  • Theta: $\Theta=\partial V/\partial t$
  • Time decay/carry.
  • Vanna: $\partial\Delta/\partial\sigma = \partial\nu/\partial S$
  • How delta changes when vol changes (or vega changes when spot moves).
  • Volga/Vomma: $\partial^2V/\partial\sigma^2$
  • Vega convexity to vol-of-vol shocks.

Useful decomposition

For short horizon: $$ \Delta P \approx \Theta\,\Delta t + \Delta\,\Delta S + \tfrac12\Gamma(\Delta S)^2 + \nu\,\Delta\sigma + ... $$

Interview Q: "What happens to vanna if spot rallies 5%?"

Correct answer template: - It depends on strike/moneyness and maturity. - ATM call tends to have vanna near zero under standard assumptions. - OTM/ITM options can have nonzero vanna; as spot moves moneyness shifts, so portfolio vanna profile changes materially.

5) "Long All Greeks" — What It Really Means

Strictly speaking

In a single plain-vanilla option, you cannot free-lunch all exposures.

What PM language often means in practice

  • Long gamma where it matters.
  • Theta not necessarily positive in pure option terms, but improved by structure/carry/relative-value.
  • Neutral-to-controlled vega.
  • Intentional second-order smile exposures (vanna/volga) via structure.

Accurate phrasing

"It is not breaking Black-Scholes identities; it is shaping net exposures across strikes/tenors using relative mispricing and constraints."

6) Buy ATM / Sell Wings and Smile Premium

Why this trade appears often

  • Wings can embed insurance demand premium.
  • Selling rich wings funds ATM convexity.

Risk

  • Short wings = short tail convexity.
  • Quiet periods harvest carry; jumps can produce severe losses.

Key lesson from SNB-style events

  • Crowd can over-sell tails due to narrative certainty.
  • A disciplined model may buy underpriced tails when market complacency is extreme.

7) Robot Strategy: Objective + Constraints (How to Explain)

Conceptual objective (important notation clarification)

w_i is usually position size (contracts/notional), not vega itself.

Two equivalent ways to write the objective:

  1. Contract-weight formulation $$ \max_w \sum_i w_i\Big[(IV_{fair,i}-IV_{mkt,i})\cdot Vega_i\Big]-\text{Costs}(w) $$ Here, vega converts IV spread into approximate dollar edge per contract.

  2. Vega-notional formulation Define $x_i := w_i\cdot Vega_i$. $$ \max_x \sum_i x_i\,(IV_{fair,i}-IV_{mkt,i})-\text{Costs}(x) $$ In this form, decision variables are vega notionals.

So: weight is not automatically vega. Vega is usually a scaling/exposure term (unless you explicitly choose vega-notional variables).

Typical constraints

  • Delta target (often near zero).
  • Gamma bounds.
  • Vega bounds or neutrality band.
  • Vanna/volga bounds.
  • Liquidity/size/notional/turnover limits.
  • Risk/capital/margin constraints.

Typical constraints in math form (contract-weight variables $w_i$)

Let each instrument $i$ have per-contract Greeks and costs: $\Delta_i,\Gamma_i,\nu_i,\text{Vanna}_i,\text{Volga}_i$, spread/liquidity metrics, and margin coefficient $m_i$.

  1. Delta neutrality / target band $$ \Delta^{L} \le \sum_i w_i\Delta_i \le \Delta^{U} $$ Often $\Delta^L=-\varepsilon_\Delta,\ \Delta^U=+\varepsilon_\Delta$.

  2. Gamma exposure control $$ \Gamma^{L} \le \sum_i w_i\Gamma_i \le \Gamma^{U} $$ For long-convexity overlays, typically $\Gamma^{L}>0$.

  3. Vega neutrality / band $$ \nu^{L} \le \sum_i w_i\nu_i \le \nu^{U} $$ Strict neutral is $\nu^L=\nu^U=0$.

  4. Vanna / Volga limits $$ ext{Vanna}^{L} \le \sum_i w_i\text{Vanna}_i \le \text{Vanna}^{U} $$ $$ ext{Volga}^{L} \le \sum_i w_i\text{Volga}_i \le \text{Volga}^{U} $$

  5. Position bounds (per instrument) $$ w_i^{\min} \le w_i \le w_i^{\max} $$

  6. Liquidity / participation cap $$ |w_i| \le \alpha_i\,ADV_i $$ or spread-aware risk budget $$ \sum_i |w_i|\,\text{SpreadCost}i \le C{\max} $$

  7. Turnover constraint (rebalance from current holdings $w_i^{old}$) $$ \sum_i |w_i-w_i^{old}| \le TO_{\max} $$

  8. Margin / capital budget $$ \sum_i m_i|w_i| \le M_{\max} $$

  9. Optional integer/cardinality constraints (listed options reality) $$ w_i \in \mathbb{Z} $$ and with binary selection variables $z_i\in{0,1}$, $$ |w_i| \le U_i z_i,\quad \sum_i z_i \le K $$ to limit number of active legs.

  10. Stress/scenario constraints (recommended in production) For each scenario $s\in\mathcal{S}$: $$ ext{PnL}_s(w) \ge -L_s $$ which caps worst-case losses under pre-defined shocks.

Why vega-neutral often used

To isolate shape-relative alpha (skew/term/wings) from broad parallel vol-level moves.

If using contract weights, vega neutrality is: $$ \sum_i w_i\cdot Vega_i \approx 0 $$ not $\sum_i w_i=0$.

Important correction

Vega-neutral does not mean guaranteed immunity to all vol moves. - Surface moves are not parallel. - Vol-of-vol and smile dynamics remain. - Model and execution error remain.

26) System Design Deep-Dive: KDB/Q vs Kafka+Flink (and HPA)

This topic is very likely to come up for a vol-overlay data platform discussion.

A) Are Kafka+Flink and KDB/Q interchangeable?

Not exactly. They optimize different layers:

  • KDB/Q: specialized time-series store + query engine (strong for tick analytics, replay, and ad-hoc time-series queries).
  • Kafka+Flink: streaming transport + stateful compute engine (strong for high-throughput processing, enrichment, and pipeline decoupling).

Interview-safe statement: "Kafka+Flink is not automatically faster than KDB/Q for every tick-data workload; they are optimized for different responsibilities."

B) Performance trade-offs (what to say)

  1. Ingestion throughput - Kafka+Flink: excellent horizontal scaling for event pipelines. - KDB/Q: also strong ingestion for market data, typically with different scaling/partition patterns.

  2. Stream processing / joins / windows - Kafka+Flink: usually better for complex stateful stream logic. - KDB/Q: can do real-time analytics but is less a general stream-orchestration framework.

  3. Tick query / replay / ad-hoc historical analytics - KDB/Q: often simpler/stronger out-of-the-box in trading environments. - Kafka+Flink alone: insufficient; requires downstream query-optimized store.

  4. Operational complexity - Kafka+Flink-only replacement is possible but typically needs extra components (query store, lakehouse, serving layer), increasing system complexity.

C) If replacing KDB/Q with Kafka+Flink, what else is needed?

At minimum: - Kafka (bus), - Flink (stateful compute), - queryable low-latency analytics store, - historical lake (Parquet/Iceberg/Delta), - hot serving cache/state layer for low-latency consumers.

So it becomes an ecosystem, not just two tools.

D) Where Kubernetes HPA fits (and where it does not)

HPA is useful for: - parsing/enrichment services, - backfill/replay workloads, - non-critical APIs, - bursty ETL/feature jobs.

HPA is usually avoided for strict hot-path loops (or used very cautiously) due to latency jitter: - per-tick risk/execution loop, - last-mile optimizer-to-execution path, - deterministic hedge engines.

Better for hot path: pre-provisioned headroom + strict SLO controls.

E) Strong final-round answer template

"I’d use a hybrid architecture: keep latency-critical risk/hedge loops deterministic with pre-sized capacity, and use Kafka+Flink where stream scalability and stateful transformations provide the most value. If replacing KDB/Q, I’d explicitly add query and historical layers, and validate p99 latency, replay determinism, and failure behavior before migration."

There is partial overlap in edge cases, but in market-maker systems these tools are commonly complementary.

34) Distributed Caching & Locking on Kubernetes (File Lock vs Redis)

A) Why OS-level file locks are risky in distributed pods

  • Pod-local file locks do not coordinate reliably across multiple pods/nodes.
  • Pod death/restart can leave uncertain lock state.
  • Networked filesystems can have lock semantics that are hard to reason about under failure.

Interview-safe statement: "I avoid pod-local file locks for distributed coordination; I use lease-based distributed locks or redesign to minimize locking."

B) Is Redis a good locking mechanism?

Yes, commonly used, but implementation details matter:

  1. lock key has TTL/lease,
  2. lock value is owner token (UUID),
  3. release uses compare-and-delete (only owner can unlock),
  4. include fencing/version where writes can race.

For stronger coordination semantics, etcd/ZooKeeper/Consul are also valid choices.

C) Your design question: data in Parquet, lock in Redis?

  • Valid for batch/nearline orchestration (dedupe, job ownership, backfill control).
  • Not ideal as a hot low-latency cache path.
  • Hot path should prefer in-memory serving and partition ownership to reduce lock contention.

D) TTL behavior explained (important)

TTL here is a lease timeout, not a job timeout.

  • If pod crashes, lease expires and another worker can take over.
  • If pod is healthy but task runs long, worker must renew/heartbeat the lease before expiry.

Typical pattern: 1. acquire lock with token + TTL, 2. start periodic renewal thread (extend TTL), 3. do work, 4. release lock only if token matches.

If renewal fails (network partition, process stalled), worker should stop writing and fail safe.

E) How to avoid false lock expiry on large writes

  1. Set conservative initial TTL based on p99 write duration.
  2. Use heartbeat renewals at short interval (e.g., TTL/3).
  3. Make writes idempotent and chunked (checkpoint progress).
  4. Add fencing token to every write so stale lock owners cannot commit late writes.
  5. Alert on near-expiry renew failures.

F) Production principle

"Use distributed leases with renewal + fencing, but reduce global locking by design (partition ownership, idempotent writes, append-only logs)."

G) Python client detail: yes, TTL/lease is set client-side

You set lease TTL directly from Python Redis client calls.

  1. Acquire lock with TTL + NX (create only if absent)
import uuid
import redis

r = redis.Redis(host="redis-host", port=6379, decode_responses=True)
lock_key = "lock:dataset:partition_2026-03-03"
token = str(uuid.uuid4())

# EX=30 means 30-second lease TTL
ok = r.set(lock_key, token, nx=True, ex=30)
if not ok:
      raise RuntimeError("lock busy")
  1. Renew/heartbeat lease while long job runs
# extend lease to 30s again (only if lock still exists)
r.expire(lock_key, 30)

# millisecond version if needed
r.pexpire(lock_key, 30000)
  1. Safe unlock (compare token then delete)
UNLOCK_LUA = """
if redis.call('get', KEYS[1]) == ARGV[1] then
   return redis.call('del', KEYS[1])
else
   return 0
end
"""
r.eval(UNLOCK_LUA, 1, lock_key, token)

Important nuance: - EXPIRE renews TTL but does not verify owner token. - For high safety, renewal can also be token-checked via Lua (same ownership check pattern as unlock). - If heartbeat fails repeatedly, worker should stop writes and fail safe.
